# 应用市场高阶技术需求文档

## 模块一：前端展示与高并发流量承载

**目标**：应对应用市场突发大流量（如爆款首发），保障系统的高可用与响应速度，深度落地 Redis 缓存策略。

* **首页榜单与应用详情查询（防缓存击穿与雪崩）**
* **业务逻辑**：用户高频访问首页热榜和热门 App 详情。
* **技术实现**：引入 `golang.org/x/sync/singleflight` 合并高并发读请求，确保单个热点失效时只有一个 Goroutine 穿透到 DB（防击穿）；在 Redis 缓存过期时间上叠加随机 Jitter 抖动值，避免大面积同时失效（防雪崩）。


* **恶意刷量防御（防缓存穿透）**
* **业务逻辑**：防御恶意脚本持续请求不存在的 App ID。
* **技术实现**：在应用层或 Redis 端构建布隆过滤器（Bloom Filter），拦截非法 ID，保护底层 PostgreSQL 免受无效查询冲击。



## 模块二：并发控制与异步任务流水线

**目标**：实战 Go 语言的并发哲学，掌握 Goroutine、Channel 与 Context 的协同调度。

* **应用下载与安装流水线（工作池与并发调度）**
* **业务逻辑**：用户触发下载后，需进行风控校验、扣减配额、生成 MinIO 临时链接、记录审计日志并发送站内信。
* **技术实现**：构建基于 Channel 的后台 Worker Pool 限制并发数；使用 Goroutine 异步处理日志与通知；通过 `context.Context` 实现风控失败时的级联取消（Cancellation）。


* **全局异步审计与优雅退出（流数据处理）**
* **业务逻辑**：无阻塞地收集并记录全站的敏感操作与关键路径日志。
* **技术实现**：通过 Go Channel 与 `time.Ticker` 实现定时定量（如满100条或满3秒）的批量入库（Bulk Insert）；捕获系统中断信号（`os/signal`），配合 `sync.WaitGroup` 确保 Channel 内残余日志刷入 DB 后实现优雅退出（Graceful Shutdown）。



## 模块三：核心交易与底层数据一致性

**目标**：榨干 PostgreSQL 的并发控制机制与底层原理，确保交易链路的绝对可靠。

* **虚拟货币流水与权益消费（防脏写与超卖）**
* **业务逻辑**：用户使用虚拟货币购买应用或兑换限时抢购的激活码。
* **技术实现**：在核心扣款事务中，使用 PostgreSQL 行级悲观锁（`SELECT ... FOR UPDATE`）保障并发一致性；手写雪花算法（Snowflake）生成全局唯一递增的流水号，并实现时钟回拨容错。


* **长链路发券与分布式锁（保障幂等性）**
* **业务逻辑**：权益发放与订单状态解耦，依赖 MQ 异步处理。
* **技术实现**：在 RocketMQ 消费端引入 Redis 分布式锁，确保重复消息消费时的绝对幂等性，杜绝重复发券；利用 Redis ZSet 或 MQ 死信队列实现未支付订单的超时自动取消。


* **秒杀与限时抢购（原子限流与库存扣减）**
* **业务逻辑**：应对运营活动的瞬时抢购流量（如抢购充值券）。
* **技术实现**：编写 Redis Lua 脚本，在单线程模型下原子化完成“校验资格 -> 限流 -> 扣减库存”，彻底消除并发竞态条件。



## 模块四：大数据量流式处理与高阶存储

**目标**：掌握大数据量场景下的内存控制、高级索引原理及 Redis 复杂数据结构。

* **高阶商品搜索与分类树（PG 高级特性）**
* **业务逻辑**：支持深层级的分类检索，以及不规则应用属性（如游戏引擎、平台支持）的多维搜索。
* **技术实现**：使用 PostgreSQL 的 `WITH RECURSIVE` 语法递归查询分类树；将动态扩展属性存入 `JSONB` 字段，并构建 `GIN` 倒排索引提升非结构化检索性能。


* **后台百万级流水导出（内存控制与流式 I/O）**
* **业务逻辑**：管理员导出海量历史交易流水报表，生成 Excel/CSV 文件。
* **技术实现**：利用 PostgreSQL 的游标（Cursor）分批次拉取数据；在 Go 端通过 `io.Pipe` 开启协程，实现边查 DB、边生成文件、边直传 MinIO 的流式处理，杜绝服务 OOM。


* **流水数据归档与冷热分离（分区表与存储优化）**
* **业务逻辑**：解决单表千万级数据导致的 B-Tree 索引变深与查询衰退问题。
* **技术实现**：引入 PostgreSQL 声明式分区表（按月或季度分区），利用优化器的分区裁剪提升查询效率；编写定时任务，将一年前的冷数据归档至 MinIO 后清理物理表。


* **开发者数据看板（Redis 高级数据结构）**
* **业务逻辑**：为开发者提供实时的应用曝光率、独立访客（UV）统计与活跃用户分析。
* **技术实现**：使用 `HyperLogLog` 以极低内存开销统计每日 UV；使用 `Bitmap` 及其位运算高效统计连续活跃用户；采用 Redis `Pipeline` 将多个指标的拉取指令打包合并，降低网络 RTT 耗时。
