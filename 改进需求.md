# 应用市场高阶技术需求文档

## 模块一：前端展示与高并发流量承载

**目标**：应对应用市场突发大流量（如爆款首发），保障系统的高可用与响应速度，深度落地 Redis 缓存策略。

* **首页榜单与应用详情查询（防缓存击穿与雪崩）**
* **业务逻辑**：用户高频访问首页热榜和热门 App 详情。
* **技术实现**：引入 `golang.org/x/sync/singleflight` 合并高并发读请求，确保单个热点失效时只有一个 Goroutine 穿透到 DB（防击穿）；在 Redis 缓存过期时间上叠加随机 Jitter 抖动值，避免大面积同时失效（防雪崩）。


* **恶意刷量防御（防缓存穿透）**
* **业务逻辑**：防御恶意脚本持续请求不存在的 App ID。
* **技术实现**：在应用层或 Redis 端构建布隆过滤器（Bloom Filter），拦截非法 ID，保护底层 PostgreSQL 免受无效查询冲击。



## 模块二：并发控制与异步任务流水线

**目标**：实战 Go 语言的并发哲学，掌握 Goroutine、Channel 与 Context 的协同调度。

* **应用下载与安装流水线（工作池与并发调度）**
* **业务逻辑**：用户触发下载后，需进行风控校验、扣减配额、生成 MinIO 临时链接、记录审计日志并发送站内信。
* **技术实现**：构建基于 Channel 的后台 Worker Pool 限制并发数；使用 Goroutine 异步处理日志与通知；通过 `context.Context` 实现风控失败时的级联取消（Cancellation）。


* **全局异步审计与优雅退出（流数据处理）**
* **业务逻辑**：无阻塞地收集并记录全站的敏感操作与关键路径日志。
* **技术实现**：通过 Go Channel 与 `time.Ticker` 实现定时定量（如满100条或满3秒）的批量入库（Bulk Insert）；捕获系统中断信号（`os/signal`），配合 `sync.WaitGroup` 确保 Channel 内残余日志刷入 DB 后实现优雅退出（Graceful Shutdown）。



## 模块三：核心交易与底层数据一致性

**目标**：榨干 PostgreSQL 的并发控制机制与底层原理，确保交易链路的绝对可靠。

* **虚拟货币流水与权益消费（防脏写与超卖）**
* **业务逻辑**：用户使用虚拟货币购买应用或兑换限时抢购的激活码。
* **技术实现**：在核心扣款事务中，使用 PostgreSQL 行级悲观锁（`SELECT ... FOR UPDATE`）保障并发一致性；手写雪花算法（Snowflake）生成全局唯一递增的流水号，并实现时钟回拨容错。


* **长链路发券与分布式锁（保障幂等性）**
* **业务逻辑**：权益发放与订单状态解耦，依赖 MQ 异步处理。
* **技术实现**：在 RocketMQ 消费端引入 Redis 分布式锁，确保重复消息消费时的绝对幂等性，杜绝重复发券；利用 Redis ZSet 或 MQ 死信队列实现未支付订单的超时自动取消。


* **秒杀与限时抢购（原子限流与库存扣减）**
* **业务逻辑**：应对运营活动的瞬时抢购流量（如抢购充值券）。
* **技术实现**：编写 Redis Lua 脚本，在单线程模型下原子化完成“校验资格 -> 限流 -> 扣减库存”，彻底消除并发竞态条件。



## 模块四：大数据量流式处理与高阶存储

**目标**：掌握大数据量场景下的内存控制、高级索引原理及 Redis 复杂数据结构。

* **高阶商品搜索与分类树（PG 高级特性）**
* **业务逻辑**：支持深层级的分类检索，以及不规则应用属性（如游戏引擎、平台支持）的多维搜索。
* **技术实现**：使用 PostgreSQL 的 `WITH RECURSIVE` 语法递归查询分类树；将动态扩展属性存入 `JSONB` 字段，并构建 `GIN` 倒排索引提升非结构化检索性能。


* **后台百万级流水导出（内存控制与流式 I/O）**
* **业务逻辑**：管理员导出海量历史交易流水报表，生成 Excel/CSV 文件。
* **技术实现**：利用 PostgreSQL 的游标（Cursor）分批次拉取数据；在 Go 端通过 `io.Pipe` 开启协程，实现边查 DB、边生成文件、边直传 MinIO 的流式处理，杜绝服务 OOM。


* **流水数据归档与冷热分离（分区表与存储优化）**
* **业务逻辑**：解决单表千万级数据导致的 B-Tree 索引变深与查询衰退问题。
* **技术实现**：引入 PostgreSQL 声明式分区表（按月或季度分区），利用优化器的分区裁剪提升查询效率；编写定时任务，将一年前的冷数据归档至 MinIO 后清理物理表。


* **开发者数据看板（Redis 高级数据结构）**
* **业务逻辑**：为开发者提供实时的应用曝光率、独立访客（UV）统计与活跃用户分析。
* **技术实现**：使用 `HyperLogLog` 以极低内存开销统计每日 UV；使用 `Bitmap` 及其位运算高效统计连续活跃用户；采用 Redis `Pipeline` 将多个指标的拉取指令打包合并，降低网络 RTT 耗时。

# 基于当前代码核对后的改进需求

以下内容基于当前仓库真实实现整理，用于区分“已经完成”“部分完成”“尚未完成”。后续对外介绍或简历包装时，应优先使用已经落地的内容，避免把 Roadmap 写成已完成能力。

## 一、RBAC 与权限缓存

### 已完成

* **带 Domain 的 Casbin RBAC**：已实现基于 Casbin 的权限校验，群组接口通过路由参数 `:id` 作为 domain；无具体 domain 的接口默认落到系统全局域 `11111111-1111-1111-1111-111111111111`。
* **本地 Enforcer 缓存池**：`common/utils/casbin_manager.go` 中使用 LRU 缓存 domain -> Enforcer，并支持 TTL 过期。
* **按需加载权限策略**：首次访问某个 domain 时，调用 `LoadFilteredPolicy(&DomainFilter{DomainID: domainID})` 只加载该域策略。
* **缓存击穿防御**：Casbin Enforcer 加载使用 `golang.org/x/sync/singleflight` 合并同一 domain 的并发加载请求。
* **RocketMQ 广播缓存失效**：支持 `invalidate_domain` 和 `purge_all` 两类消息，多实例收到广播后驱逐本地权限缓存。

### 部分完成 / 需要修正表述

* **Market 应用资源权限**：Market 路由使用 `:app_id`，当前 Casbin 中间件只识别 `:id` 作为 domain，因此应用编辑、下架、删除的 Casbin 校验走全局域权限，应用所有者校验在业务 handler 中补充完成。后续如需更细粒度资源权限，需要扩展 domain/resource 解析策略。
* **权限变更广播覆盖面**：角色权限变更已本地 `PurgeAll` 并广播 `purge_all`。群成员变更是否完整广播 `invalidate_domain`，需要逐个核对 IAM group service 并补齐测试。

### 待改进

* 为 `CasbinRequire` 增加可配置 domain 解析策略，支持从 `:id`、`:group_id`、`:app_id` 或业务查询结果中解析资源域。
* 为权限缓存失效补集成测试：角色权限变更、成员加入、成员移除、成员角色变更、多实例广播消费。
* 记录权限缓存命中率、加载耗时、singleflight 合并次数等指标，便于压测和线上观测。

## 二、Redis 缓存层与热点保护

### 已完成

* **Redis Session**：已用于 JWT + Redis Session 混合认证、自动续签和顶号判定。
* **Redis ZSET 下载热榜**：Market 下载链路已使用 Redis ZSET 维护 daily、weekly、total 下载榜单，并使用 Pipeline 更新多个榜单。
* **Redis 发布锁**：Market 发布应用时使用 Redis `SetNX` 对同一开发者同一幂等键做并发互斥。
* **Redis 滑动窗口限流**：代码支持 Redis Lua 滑动窗口限流，但默认配置为 `memory` 后端。
* **RedisLock 工具类**：已有 UUID value + Lua compare-and-delete 解锁的分布式锁封装。

### 尚未完成

* **首页榜单缓存层**：目前热榜实时数据来自 Redis ZSET，但没有针对“首页聚合榜单响应体”的 cache-aside 缓存层。
* **应用详情 Redis 缓存**：`GET /api/v1/market/apps/:app_id` 当前直接查 PostgreSQL，未接 Redis 缓存。
* **热点 key singleflight 回源合并**：`singleflight` 当前只用于 Casbin Enforcer 加载，未用于应用详情或首页缓存回源。
* **随机 TTL 防雪崩**：当前未看到 Redis 缓存 TTL jitter 实现。
* **缓存空值 / Bloom Filter 防穿透**：当前没有对不存在 app_id 的空值缓存或 Bloom Filter。

### 待改进

* 为应用详情增加 cache-aside：`market:app:detail:{app_id}`，缓存命中直接返回，miss 时 singleflight 合并 DB 回源。
* 为首页/榜单聚合增加缓存：例如 `market:home:v1` 或按榜单维度拆分 key。
* 缓存 TTL 采用基础 TTL + 随机 jitter，例如 `5m + rand(0~60s)`。
* 对不存在 app_id 做短 TTL 空值缓存，降低恶意穿透。
* 如 app_id 数据量较大且查询不存在 ID 很频繁，再考虑 Bloom Filter；否则先用参数校验 + 空值缓存即可。
* 增加缓存命中率、回源耗时、singleflight shared 次数等监控指标。

## 三、监控、压测与性能数据

### 已完成

* **Prometheus + cAdvisor**：已通过 Prometheus scrape cAdvisor，采集容器 CPU、内存、网络、磁盘等指标。
* **k6 remote write**：k6 容器已配置 `K6_PROMETHEUS_RW_SERVER_URL=http://prometheus:9090/api/v1/write`，压测指标写入 Prometheus。
* **docker stats 归档**：`run-load-tests.sh` 会为每个压测 case 采样 `docker stats` 并输出 CSV。
* **Loki + Alloy 日志链路**：Alloy 发现 Docker 容器并采集 stdout 日志，写入 Loki。
* **Grafana provisioning**：已配置 Prometheus 和 Loki datasource，并加载 Docker/Loki 相关 dashboard。
* **请求级 TraceID**：HTTP 请求支持 `X-Trace-ID`，响应头和错误响应中会回写 request_id，请求日志也记录 trace_id。
* **功能开关压测**：压测脚本支持开关 JWT、Casbin、RateLimit、RequestPool，用于对比治理能力对性能的影响。

### 部分完成 / 需要修正表述

* **压测指标不是推到 Loki**：k6 指标实际通过 remote write 写入 Prometheus；Loki 存储应用和容器日志。Grafana 可同时查询 Prometheus 指标与 Loki 日志。
* **Prometheus 未直接采应用指标**：当前 Prometheus 配置只 scrape `prometheus` 和 `cadvisor`，应用没有 `/metrics` endpoint。
* **链路追踪是请求 ID，不是完整分布式追踪**：当前是 `X-Trace-ID` 请求级追踪，没有 OpenTelemetry span。
* **熔断中间件未接入压测开关**：`common/middleware/broker.go` 有简单熔断器实现，但未看到在 `main.go` 或路由中实际挂载，也未进入 `run-load-tests.sh` 的开关矩阵。
* **性能数字需要压测报告支撑**：如“落库接口 QPS 50 P99 50ms”“缓存命中 QPS 700 P99 30ms”“快速失败 P99 50us”等，当前代码仓库无法直接证明，必须绑定具体接口、环境、开关、数据规模和结果文件。

### 待改进

* 接入 `prometheus/client_golang`，为 App 暴露 `/metrics`。
* 增加 HTTP middleware 指标：请求数、错误数、latency histogram、按 path/method/status 聚合。
* 暴露 Go runtime 指标：goroutine 数、GC、heap、CPU。
* 暴露 DB/Redis 客户端池指标：`sql.DB.Stats()`、go-redis `PoolStats()`。
* 暴露请求池指标：池容量、运行任务数、拒绝数、panic 数。
* 将熔断器真正接入指定路由或依赖调用链，并加入压测开关。
* 增加 pprof，用于 CPU、heap、goroutine、mutex、block profile 定位。
* 建立标准压测报告模板：接口、版本、环境、功能开关、QPS、P50/P95/P99、错误率、CPU/内存、DB/Redis/MinIO 状态、结论。

## 四、交易链路、权益发放与幂等

### 已完成

* **虚拟钱包流水**：已实现充值、扣款、转账、流水查询。
* **订单购买链路**：已实现订单创建、支付、取消、一步购买。
* **订阅发放**：订单支付成功后创建 `subscriptions` 记录。
* **并发扣款控制**：钱包扣款在 DB transaction 内使用 `SELECT FOR UPDATE` 锁钱包行，避免并发超扣。
* **转账死锁规避**：转账时按 wallet_id 字符串排序后加锁，避免 AB/BA 死锁。
* **幂等约束**：`wallet_transactions.idempotency_key` 唯一；`orders.idempotency_key` partial unique index；Market 发布也有开发者 + 幂等键表达式唯一索引。

### 尚未完成

* **权益发放 RocketMQ 异步解耦**：当前订阅发放在订单支付事务内同步完成，没有看到 RocketMQ 异步权益发放链路。
* **消费端 Redis 分布式锁发券幂等**：已有 RedisLock 工具类，但没有看到权益/发券消费者使用 Redis 分布式锁。
* **工单链路**：当前代码中是订单 `orders`，没有看到“工单”概念。
* **未支付订单超时自动取消**：当前没有看到 Redis ZSET 延迟队列或 MQ 死信队列实现订单超时取消。
* **秒杀库存扣减 Lua**：当前没有活动库存、资格校验、Redis Lua 扣库存链路。

### 待改进

* 如需异步权益发放，可将订单支付拆为：支付成功 -> 写 outbox / 发 RocketMQ 消息 -> 消费端幂等创建权益。
* 消费端幂等建议同时依赖 DB 唯一键和 Redis 分布式锁，Redis 锁只做并发优化，DB 约束做最终兜底。
* 增加订单超时取消 worker 或延迟消息机制。
* 若引入优惠券/激活码秒杀，再设计库存表、资格表、Redis Lua 预扣库存和最终 DB 事务确认。

## 五、大数据量导出与流式处理

### 已完成

* **审计 CSV 流式导出**：`common/audit/export.go` 使用 `io.Pipe` 边生成 CSV 边通过 HTTP 返回。
* **Keyset cursor 分页**：审计导出使用 `created_at,event_id` 作为稳定游标，避免深 offset。
* **批量查询**：每批默认 500 条，降低一次性内存占用。

### 部分完成 / 需要修正表述

* **不是 PostgreSQL server-side cursor**：当前是应用层 keyset cursor，不是 `DECLARE CURSOR/FETCH` 的 PostgreSQL 游标。
* **没有边上传 MinIO**：当前导出是边查、边写、边返回 HTTP 响应；没有看到边写边上传 MinIO。
* **导出对象是 audit_events**：当前不是钱包流水百万级导出，而是审计事件导出。

### 待改进

* 为钱包流水、订单流水、审计事件分别提供大批量导出接口。
* 如果需要异步导出文件，使用 `io.Pipe` 将 CSV writer 与 MinIO `PutObject` 串起来，实现边查边上传。
* 增加导出任务表，记录状态、进度、文件 object_name、失败原因。
* 为审计导出补 `(created_at, event_id)` 组合索引，匹配 keyset cursor。
* 用 cAdvisor 和 pprof 对不同 batch size 的内存峰值进行对比，形成压测报告。
