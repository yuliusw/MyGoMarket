# RPA-market 八股总纲

本文基于 `ai-task.md` 的当前工程事实整理。回答项目经历、架构设计、技术亮点和面试追问时，以本文为主线；具体 Go、Redis、PostgreSQL、监控压测细节可继续参考 `doc/八股/` 下的专题文档。

## 1. 项目一句话介绍

`RPA-market` 是一个 Go + Vue 3 的 RPA 应用市场 Monorepo。后端以 Gin HTTP 服务为主，同进程启动 gRPC 服务；业务覆盖 IAM 认证授权、应用市场、钱包、订单支付、权益发放和审计导出；基础设施包含 PostgreSQL、Redis、MinIO、RocketMQ、Casbin、Prometheus、Loki、Grafana 和 k6。

## 2. 技术栈

- 后端：Go、Gin、GORM、gRPC、Viper、Zap。
- 数据库：PostgreSQL，使用事务、行级锁、唯一索引、JSONB、分区表和触发器。
- 缓存与状态：Redis，用于 Session、限流可选后端、排行榜、缓存、分布式锁和订单/权益 worker 互斥。
- 对象存储：MinIO，用于应用包、审计导出文件和日志存储后端。
- 权限：Casbin Domain RBAC，配合 LRU 缓存、singleflight 和 RocketMQ 广播失效。
- 消息：RocketMQ，当前主要用于 Casbin 权限缓存跨实例同步。
- 前端：Vue 3、Vite、TypeScript，路由显式配置，后端请求统一封装在 `src/api.ts`。
- 可观测性：Prometheus、cAdvisor、Loki、Alloy、Grafana、k6、应用 `/metrics`。

## 3. 启动链路

后端入口是 `main.go` 和 `app.go`，初始化顺序必须按当前代码事实理解：

1. `config.InitConfig` 加载配置。
2. `database.InitGORM` 初始化 PostgreSQL/GORM。
3. `database.InitRedis` 初始化 Redis。
4. `database.InitMinio` 初始化 MinIO。
5. `rocketmq.Init` 初始化 RocketMQ。
6. `utils.InitCasbinPool` 初始化 Casbin Enforcer 缓存池。
7. `iam.RegisterHandlers` 注册 IAM 路由。
8. `market.RegisterMarketHandlers` 注册 Market 路由。

HTTP 默认端口由 `server.port` 控制，当前为 `:12660`。gRPC 默认端口为 `:12661`，受 `grpc.enabled` 和 `grpc.port` 控制。当前仍是单体仓库，gRPC 只是模块边界和未来拆分准备，不要把项目讲成已经微服务化，也不要引入动态服务发现口径。

## 5. 认证与会话

项目采用 JWT + Redis Session 混合双检。

- 登录成功必须把 Session 写入 Redis，写失败直接返回 `503`。
- JWT 使用 `HS256`，浏览器端通过 `auth_token` Cookie 携带，也支持 `Authorization: Bearer <token>`。
- Cookie 模式会校验 Redis 里的 `session_id`，用于防顶号和自动续签。
- 纯 API 客户端只携带有效 Bearer token 也可通过，不再强制要求 `session_id` Cookie。
- JWT 过期后的自动续签仍依赖 Redis Session 和 Cookie 一致。
- 登出和修改密码会同时清理 `auth_token` 与 Redis `session_id`。

面试回答重点：不是纯无状态 JWT，而是服务端可控的混合会话方案。好处是支持登出失效、改密失效、顶号控制和续签；代价是 Redis 成为认证强依赖。

## 6. 功能开关、限流、请求池与熔断

`features` 配置项支持 JWT、Casbin、CORS、限流、请求池、熔断器、订单超时取消等能力的开关和参数配置，并支持环境变量覆盖。

- 限流支持 `memory` 和 `redis` 两种后端。
- Market 公开接口、登录、注册路由已经挂载限流器。
- 默认限流后端仍是 `memory`，如果要压测 Redis 滑动窗口限流，需要显式设置 `features.rate_limit.backend=redis`。
- 全局请求池已接入，满载时快速拒绝并返回 `503`。
- 熔断器通过 `ConfiguredCircuitBreaker()` 挂载到全局 Gin 链路，配置项是 `features.circuit_breaker.enabled/max_failures/timeout_seconds`。
- 绕过认证压测时，`auth_bypass_user_id` 必须是真实存在的 UUID，默认管理员 ID 是 `00000000-0000-0000-0000-000000000001`。

面试回答重点：限流解决单用户或单 key 频率问题，请求池解决整体并发过载问题，熔断器解决下游持续失败时快速失败问题。

## 7. Casbin 权限与 RBAC

项目使用带 Domain 的 Casbin RBAC 模型，配置路径是 `config/casbin/RBAC.conf`。

- 群组接口通过路由参数 `:id` 识别 Domain。
- Market 应用接口可通过 `:app_id` 识别资源域。
- 无明确 Domain 的接口回落到系统全局域 `11111111-1111-1111-1111-111111111111`。
- Casbin Manager 使用 `LRU 内存缓存池 + singleflight 并发抑制 + TTL 过期`。
- 首次访问某 Domain 时按需 `LoadFilteredPolicy`，不是全量加载所有策略。
- 角色或权限变更后，清理本地缓存并通过 RocketMQ 广播失效消息。
- 支持 `invalidate_domain` 和 `purge_all` 两类失效指令。

已开放的角色权限接口：

```text
GET /api/v1/iam/roles
GET /api/v1/iam/roles/:role_id
GET /api/v1/iam/permissions
PUT /api/v1/iam/roles/:role_id/permissions
```

Market 应用的编辑、下架、删除在 Casbin 层可按 `:app_id` 应用域校验，业务 Handler 仍保留 `developer_id` 所有者比对作为最终兜底。

## 8. PostgreSQL 设计

当前主数据库初始化脚本是 `script/init_better.sql`，`script/init.sql` 已归档。

已落地能力：

- 业务表索引、分区定义、触发器、审计表、下载指标表。
- MinIO 删除补偿表 `minio_delete_retries`。
- 订单表 `orders`。
- 权益本地消息表 `entitlement_outbox`。
- 新建群组创建者默认角色已从 `superadmin` 修正为 `owner`。

核心数据库能力：

- 钱包扣款、充值、转账使用事务和 `SELECT ... FOR UPDATE`。
- 订单支付锁订单行，钱包扣款锁钱包行。
- 转账前按两个 `wallet_id` 字符串排序后依次加锁，避免 AB/BA 死锁。
- 幂等靠业务查询和唯一索引双重兜底，例如订单幂等、钱包流水幂等、Market 发布幂等。
- 订阅表按过期时间分区，权益发放通过唯一索引防重发。

严禁通过生成 DDL/DML 清理当前数据库测试数据。面试可以讲数据一致性，但不要把清库脚本当成常规操作。

## 9. Market 核心链路

标准 API：

```text
GET    /api/v1/market/apps
GET    /api/v1/market/apps/:app_id
GET    /api/v1/market/apps/:app_id/download
GET    /api/v1/market/rankings
POST   /api/v1/market/apps
PUT    /api/v1/market/apps/:app_id
PUT    /api/v1/market/apps/:app_id/offshelf
DELETE /api/v1/market/apps/:app_id
```

发布链路亮点：

- 严格校验 `.zip`、`.gz`、`.tgz` 后缀和对应文件头魔数。
- MinIO ObjectName 使用服务端生成的纯 UUID 路径。
- 原始文件名只存 DB Metadata，并做路径安全与 UTF-8 规范化处理。
- 支持 `Idempotency-Key` 和 `idempotency_key` 幂等检测。
- Redis `SetNX` 短锁加 DB 唯一索引双重防并发重复发布。
- DB 写入失败时同步触发 MinIO 对象删除补偿并记录审计日志。

变更和删除链路：

- 更新、下架、删除严格校验 `developer_id`。
- 更新数据库行时使用 GORM `Updates` 字段级更新，避免 `Save` 并发覆盖。
- 删除 MinIO 对象失败会写入 `minio_delete_retries`，由后台 Worker 异步重试。

下载和统计链路：

- 下载接口签发 5 分钟 MinIO 预签名 URL。
- HTTP 返回 `307` 重定向，不由业务服务代理大文件。
- 同步记录 `app_download_metrics` 持久化指标。
- Redis Pipeline 批量累加 daily、weekly、total 三类 ZSET 排行榜。
- 默认热榜缓存会做短 TTL 失效。

缓存治理：

- 应用详情 `GET /apps/:app_id` 已采用 Cache-Aside。
- Redis key 是 `market:app:detail:{app_id}`。
- Cache Miss 通过 `singleflight` 合并并发回源。
- 正常 TTL 是 `5m + rand(0~60s)`。
- 查无实体写入 30 秒空值标记防穿透。
- 发布、更新、下架、删除后主动删除详情缓存。
- 排行榜接口有 `market:rank:cache:{type}:{limit}` 结构体短缓存，默认 10 秒。

## 10. Wallet、Order 与权益发放

gRPC 契约：

- `proto/wallet/v1/wallet.proto` 和 Buf 构建配置已就绪。
- Go 存根输出到 `gen/go`。
- `WalletService` 暴露 `GetWallet` 和 `GetOrCreateWallet`。
- `WalletRepository` 支持按 `owner_type + owner_id + currency_code` 查询或初始化钱包。

HTTP 端点：

- 钱包：`/api/v1/wallets/me`、流水列表、充值、扣款、转账。
- 订单：purchase、创建、列表、详情、支付、取消。
- 订单状态枚举：`pending`、`paid`、`cancelled`。

交易一致性：

- 钱包扣款与订单支付在同一个 DB Transaction 内执行。
- 事务中先 `SELECT ... FOR UPDATE` 锁住钱包行。
- `DebitInTx` 扣款后写流水，`reference_id` 存 `order_id`。
- 订单状态改为 `paid`，并写入 `entitlement_outbox`。
- 事务提交后后台 Worker 异步发放 `subscriptions` 权益。
- 发放完成后回填 `orders.subscription_id`。
- 权益发放 worker 基于 RedisLock 按 `order_id` 加分布式互斥锁。
- 订阅分区表用 `(user_id, app_id, source_order_id)` 唯一索引防重发。

重要口径：当前系统只有订单 `Order`，不存在“工单 Work Order”。当前没有库存模型，所以未实现 Redis Lua 库存预扣减。

## 11. 审计、错误响应与生命周期

统一错误响应：

- 项目统一使用 `common/response` 响应。
- 代码库中无裸露的 `gin.H{"error": ...}`。
- 请求链路自动注入 `X-Trace-ID`。
- 异常响应体中的 `request_id` 复用 TraceID。

审计日志：

- Market 审计使用 `Channel + Ticker` 异步内存攒批落库。
- 审计 CSV 导出在 `common/audit/export.go`。
- 导出基于 `io.Pipe` 实现边查边写边返回 HTTP 响应。
- 查询使用 Keyset 游标分页，游标组合键为 `created_at, event_id`。
- 单批查询 500 条。
- 审计导出支持 HTTP 流式响应和 MinIO 双写转存。
- MinIO ObjectName 前缀是 `audit-exports/`，响应头返回 `X-MinIO-Object`。

生命周期：

- `main.go` 使用标准 `http.Server` 实现 `SIGINT/SIGTERM` 优雅停机。
- 资源释放顺序是关闭 HTTP、停止请求池、停止 MinIO 删除重试 Worker 和订单异步 Worker、Flush 审计日志、断开 RocketMQ、关闭 Redis、关闭 PostgreSQL。

面试回答重点：审计是旁路异步能力，优先保护主链路延迟；如果强审计不能丢，需要改成 MQ 或 Outbox Pattern。

## 12. 大数据量导出

审计导出目前只支持 `audit_events` 表。钱包流水和订单流水可以通过 Admin 查询接口读取，但尚未提供 CSV 导出。

设计亮点：

- `io.Pipe` 避免把完整 CSV 放入内存。
- HTTP 端边写边返回，客户端可流式接收。
- Keyset 游标分页避免 offset 深分页性能退化和并发插入导致的重复/跳过。
- MinIO 双写转存方便归档和二次下载。

注意：当前是应用层 Keyset 内存游标，不是 PostgreSQL 服务端 `DECLARE CURSOR`。

## 15. 可以重点包装的项目亮点

- 混合认证：JWT + Redis Session，兼顾短 token 和服务端主动失效。
- 权限体系：Casbin Domain RBAC + LRU 缓存 + singleflight + RocketMQ 广播失效。
- 上传安全：扩展名、文件头、原始文件名、UTF-8、服务端对象路径、SHA256、ETag 多层校验。
- 跨存储一致性：MinIO 与 PostgreSQL 非事务场景通过幂等、补偿和审计兜底。
- 实时排行榜：Redis ZSET + Pipeline 做实时 TopN，PostgreSQL 指标做持久事实源。
- 缓存治理：应用详情 Cache-Aside、singleflight 防击穿、TTL 抖动、空值缓存防穿透。
- 交易一致性：订单、钱包扣款、流水、Outbox 在同一事务内提交。
- 异步权益：支付事务只写 `entitlement_outbox`，后台 Worker 发放订阅并用唯一索引兜底防重。
- 并发控制：钱包行锁、订单行锁、转账固定锁顺序、RedisLock 分布式互斥。
- 过载保护：限流、全局请求池快速失败、熔断器组合保护。
- 流式导出：`io.Pipe` + Keyset 分页 + MinIO 双写。
- 可观测性：Prometheus、Loki、Grafana、k6、TraceID 和 `/metrics` 形成基础闭环。
- 优雅退出：HTTP、worker、审计、MQ、Redis、PostgreSQL 有明确释放顺序。

## 17. 当前风险与优化方向

- Redis Session 是认证强依赖，高并发要关注 Redis QPS、连接池和故障策略。
- 请求池在 Gin handler 外层做全局并发限制，需持续通过压测和 race 检查确认安全性。
- Market 发布锁当前是短锁思路，最终一致性依赖 DB 唯一索引和补偿兜底。
- MinIO 删除补偿 worker 多实例部署需要任务抢占或分布式互斥，避免重复处理。
- 审计内存队列在进程崩溃时可能丢失，强审计场景应改 MQ 或 Outbox。
- 当前没有库存模型，也没有 Redis Lua 库存预扣减，不要在项目介绍中误讲。
- 链路追踪仍停留在 TraceID 日志串联阶段，尚未生成 OpenTelemetry 标准 Span。
- gRPC 当前主要是模块边界准备，不代表项目已经完成微服务拆分。
