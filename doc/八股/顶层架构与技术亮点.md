# RPA Market 顶层架构与技术亮点八股

本文基于当前仓库顶层文档和关键实现代码整理，先从顶层架构、主调用链和项目可讲亮点入手。后续可以继续细化到 IAM、Market、Wallet、Order、Admin/gRPC、前端等单模块专题。

## 1. 项目一句话介绍

这是一个面向 RPA 应用市场的 Go + Vue 单体仓库。后端以 Gin HTTP 服务为主，同时启动 gRPC 服务；业务按 IAM、Market、Wallet、Order、Admin 等域拆分；基础设施包含 PostgreSQL/GORM、Redis、MinIO、RocketMQ、Casbin、ants 协程池、异步审计、补偿重试、限流、请求池快速失败和优雅退出。

前端是独立的 Vue 3 + TypeScript + Vite 应用，主要通过 Axios 调用后端 REST API。

## 2. 顶层目录职责

- `main.go`：后端统一启动入口。负责初始化日志、配置、PostgreSQL、Redis、MinIO、RocketMQ、Casbin 同步、HTTP 路由、gRPC 服务和优雅退出。
- `app.go`：应用依赖聚合入口。初始化 JWT 中间件依赖、IAM Group/Role/User service 依赖。
- `config/`：运行配置和 Casbin RBAC 模型。
- `common/`：公共基础设施，包括配置、数据库、缓存、对象存储、中间件、审计、gRPC server、消息队列、JWT、Casbin、协程池等。
- `services/`：业务服务目录，核心包括 `iam`、`market`、`wallet`、`order`、`admin`。
- `proto/`：gRPC proto 定义，目前有 `admin/v1`、`wallet/v1`。
- `gen/go/`：protobuf 生成的 Go 代码。
- `frontend/RPA-market/`：Vue 3 前端工程。
- `script/`：本地 Docker Compose、数据库初始化、Redis、RocketMQ、Prometheus、Loki、Grafana、k6 压测脚本等。
- `deployments/`：Kubernetes/Istio 部署资源预留。

## 3. 后端技术栈

- HTTP 框架：`github.com/gin-gonic/gin`。
- ORM：`gorm.io/gorm` + `gorm.io/driver/postgres`，数据库是 PostgreSQL。
- Redis：`github.com/redis/go-redis/v9`，用于 Session、限流可选后端、排行榜 ZSET、上传并发锁等。
- 对象存储：`github.com/minio/minio-go/v7`，用于应用包、头像等文件存储，下载使用预签名 URL。
- 权限：`github.com/casbin/casbin/v3`，自定义 DB adapter，按 domain 懒加载 Enforcer。
- 消息队列：`github.com/apache/rocketmq-client-go/v2`，用于 Casbin 缓存跨实例同步；仓库也保留了 Kafka 封装。
- 协程池：`github.com/panjf2000/ants/v2`，用于请求池快速失败和通用/IO 池。
- JWT：`github.com/golang-jwt/jwt/v5`。
- UUID：`github.com/google/uuid`。
- 精确金额：`github.com/shopspring/decimal`，用于钱包和订单金额。
- 配置：`github.com/spf13/viper`。
- 日志：`go.uber.org/zap`。
- gRPC：`google.golang.org/grpc` + `google.golang.org/protobuf`。
- 并发防击穿：`golang.org/x/sync/singleflight`，用于 Casbin Enforcer 加载合并。
- LRU 缓存：`github.com/hashicorp/golang-lru/v2`，用于 Casbin domain Enforcer 缓存。

## 4. 后端启动链路

入口在 `main.go`。

1. 初始化日志：`logs.InitLogger()`。
2. 加载配置：`config.InitConfig("config.yaml")`。
3. 初始化数据库：`database.InitGORM()`。
4. 启动异步审计 writer：`audit.Start(database.DB)`。
5. 初始化 Redis：`database.InitRedis()`。
6. 初始化 MinIO：`database.InitMinio()`。
7. 启动 MinIO 删除补偿 worker：`audit.StartMinioDeleteRetryWorker(database.DB, database.GlobalMinio)`。
8. 初始化 RocketMQ：`rocketmq.Init()`。
9. 初始化 Casbin 同步：`initAuthorizationSync()`。
10. 聚合初始化 IAM 依赖：`RegisterService()`。
11. 创建 Gin：`gin.Default()`。
12. 挂全局中间件：`middleware.GinLogger()`、`middleware.ConfiguredRequestPoolFastFail()`。
13. 注册业务路由：IAM、Market、Order、Wallet、Admin。
14. 创建 HTTP server，默认端口 `12660`。
15. 创建并启动 gRPC server，默认端口 `12661`。
16. 监听 `SIGINT` / `SIGTERM`，执行优雅退出。

需要注意的点：

- HTTP 和 gRPC 同进程启动，任一 server 启动失败都会通过 `serverErr` 反馈。
- gRPC server 默认由配置 `grpc.enabled` 控制。
- 优雅退出会依次停止 HTTP、gRPC、MinIO retry worker、审计 writer、协程池、RocketMQ、Redis、PostgreSQL。
- `serverErr` 同时接收 HTTP 和 gRPC 的退出结果，如果某个 server 正常返回 `nil`，主协程会提前从 select 返回；这类并发生命周期管理后续可以再重点审查。

## 5. 配置模型

核心配置在 `config/config.yaml`。

- `server.port`：HTTP 端口，默认 `12660`。
- `server.shutdown_timeout_seconds`：优雅退出超时，默认 `15` 秒。
- `database.*`：PostgreSQL 连接、连接池和日志级别。
- `redis.*`：Redis 地址、连接池、读写超时。
- `minio.*`：MinIO endpoint、bucket、鉴权信息。
- `rocketmq.addrs`：RocketMQ nameserver 地址。
- `casbin.model_path`：Casbin RBAC 模型路径。
- `casbin.cache_ttl_seconds`：Enforcer 缓存 TTL。
- `grpc.enabled` / `grpc.port`：gRPC 开关和端口。
- `features.jwt_auth`：JWT 鉴权开关。
- `features.casbin_authz`：Casbin 授权开关。
- `features.cors`：CORS 开关。
- `features.rate_limit`：限流配置，支持 memory 或 redis 后端。
- `features.request_pool`：HTTP 请求池配置。

雷点：

- 本地 `config/config.yaml` 和 `script/config.yaml` 都可能存在，容器内真实配置要看部署脚本挂载或拷贝路径。
- 部署文档强调真实密码通过 `script/.env` 注入，不应提交 `.env`。
- 功能开关存在 `OptionalJWTAuth` / `OptionalCasbinRequire` 这类可选中间件，讲权限时要说明“配置打开时才生效”。

## 6. HTTP 中间件链路

全局中间件：

- `GinLogger()`：请求日志与 trace 信息。
- `ConfiguredRequestPoolFastFail()`：将请求处理提交到 ants 请求池，池满直接返回 `503 SERVER_BUSY`。

路由组中间件：

- 公开接口通常挂 `ConfiguredRateLimit()`，例如 IAM 注册/登录、Market 列表/详情/下载/热榜。
- 私有接口通常挂 `OptionalJWTAuth()` + `OptionalCors()`。
- 权限敏感接口额外挂 `OptionalCasbinRequire("permission")`。

调用顺序示例：

1. 请求进入 Gin。
2. 全局日志中间件记录请求。
3. 全局请求池中间件尝试提交任务。
4. 路由组限流或 JWT 鉴权。
5. Casbin 权限校验。
6. 业务 handler。
7. 统一 response/error 返回。

雷点：

- `RequestPoolFastFail` 在中间件里 `Submit(func(){ c.Next() })` 后等待 `done` channel。这里用到了 Go channel 做同步，保证请求处理完成后中间件再返回。
- Gin 的 `Context` 被放到另一个 goroutine 中执行，虽然外层同步等待，但这属于比较敏感的用法。面试可解释为“通过请求池限制并发并快速失败”，但也要知道 Gin 官方通常不建议异步 goroutine 直接使用原始 `Context`。
- 请求池使用 `ants.WithNonblocking(true)`，池满时不会阻塞排队，而是直接返回过载错误。

## 7. 认证链路

实现位置：`common/middleware/auth.go`、`services/iam/app/user_auth.go`、`services/iam/repository/user_repository.go`。

登录后的状态模型：

- 短期凭证：JWT，写入 `auth_token` HttpOnly Cookie，也可通过 `Authorization: Bearer` 携带。
- 长期会话：`session_id`，写入 Cookie，同时存 Redis。
- Redis Session 用于自动续签和顶号判定。

JWTAuth 流程：

1. 从 Header 或 `auth_token` Cookie 取 JWT。
2. 严格校验 JWT。
3. JWT 有效时，再校验 Redis 中的 Session 是否等于 Cookie 的 `session_id`。
4. JWT 过期时，用 `ParseTokenIgnoreExpiry` 提取 user_id。
5. Redis Session 仍有效且一致时，签发新 JWT 并刷新 Redis Session TTL。
6. Session 不存在、不一致或 token 无效时返回 `401`。

可讲亮点：

- 不是纯无状态 JWT，而是 JWT + Redis Session 的混合方案。
- 支持短 token 自动续签。
- 支持顶号判定，同一用户 Redis 中只保留当前有效 session。
- 修改密码后会清理 Session 和 Cookie，强制重新登录。

雷点：

- 非浏览器客户端如果只带 Bearer token，不带 `session_id` Cookie，当前校验会失败。
- Cookie 的 `Secure` 当前代码里是 `false`，生产 HTTPS 环境应调整。
- JWT 有效时仍查 Redis，因此 Redis 不可用会导致私有接口不可用。

## 8. 授权链路：Casbin + Domain + RocketMQ 同步

实现位置：`common/middleware/authz.go`、`common/utils/casbin_manager.go`、`common/utils/casbin_adapter.go`、`common/queue/rocketmq/casbin.go`。

权限模型：

- 使用 Casbin RBAC。
- 权限按 domain 隔离，群组 ID 是一个 domain。
- 没有具体 group 路由参数时，默认使用系统全局域 `11111111-1111-1111-1111-111111111111`。
- 权限码包括 `app:create`、`app:edit`、`group:view`、`role:manage` 等。

CasbinRequire 流程：

1. 从 Gin context 取当前 `user_id`。
2. 从路由参数 `:id` 获取 group/domain ID。
3. 没有 `:id` 时使用全局域。
4. 从 `utils.EnforcerPool.GetEnforcer(domainID)` 获取 Enforcer。
5. 调用 `enforcer.Enforce(userID, groupID, requirePermission)`。
6. 不通过返回 `403`。

缓存设计：

- `CasbinManager` 用 `hashicorp/golang-lru/v2` 缓存 domain -> Enforcer。
- 每个缓存项带 `loadedAt`，超过 TTL 后删除重载。
- `singleflight.Group` 合并同一个 domain 的并发加载，避免缓存击穿。
- `LoadFilteredPolicy(&DomainFilter{DomainID: domainID})` 只加载当前域策略，不全量加载所有策略。

跨实例同步：

- 角色权限或群成员权限变化后，业务层发布 RocketMQ 消息。
- 消息动作包括 `invalidate_domain` 和 `purge_all`。
- 消费者使用广播消费，所有实例都能收到并驱逐本地缓存。

雷点：

- Market 路由里的 `:app_id` 并不是 Casbin 中间件识别的 `:id`，因此 `app:edit` 等权限当前按全局域校验，应用所有者校验在业务 handler 中补充完成。
- 如果忘记发布 Casbin 同步消息，多实例场景会出现权限缓存短期不一致，只能等 TTL 过期。
- `purge_all` 用于全局 P 规则变化，成本比单 domain 驱逐更高。

## 9. Market 应用市场调用链

实现位置：`services/market/router.go`、`services/market/app/market.go`、`services/market/respository/app_respository.go`。

公开接口：

- `GET /api/v1/market/apps`：分页应用列表。
- `GET /api/v1/market/apps/:app_id`：应用详情。
- `GET /api/v1/market/apps/:app_id/download`：下载应用。
- `GET /api/v1/market/rankings`：下载热榜。

私有接口：

- `POST /api/v1/market/apps`：发布应用。
- `PUT /api/v1/market/apps/:app_id`：更新应用。
- `PUT /api/v1/market/apps/:app_id/offshelf`：下架应用。
- `DELETE /api/v1/market/apps/:app_id`：删除应用。

发布应用链路：

1. JWT 获取当前开发者 `user_id`。
2. 绑定 multipart 表单字段。
3. 校验上传文件名，禁止路径分隔符、NUL、不合法 UTF-8。
4. 限制文件大小，当前最大 100MB。
5. 解析 `Idempotency-Key` 请求头或表单字段。
6. 查询 DB 是否已有同开发者同幂等键记录，有则返回原 `app_id`。
7. 使用 Redis `SetNX` 获取 `market:publish:lock:{developer_id}:{idempotency_key}` 并发锁。
8. 校验扩展名，仅允许 `.zip`、`.gz`、`.tgz`。
9. 先保存到本地临时文件。
10. 读取文件头，用 `http.DetectContentType` 校验真实内容类型。
11. 计算 SHA256。
12. 服务端生成 MinIO object name：`apps/{developer_id}/{app_id}/app{ext}`。
13. 上传到 MinIO。
14. 组装 metadata，包括原始文件名、object name、size、content type、sha256、etag、version、idempotency_key。
15. 写入 DB apps 表。
16. 如果 DB 写入失败，尝试删除已上传 MinIO 对象做补偿。
17. 写异步审计事件。

下载链路：

1. 校验 `app_id`。
2. 查询 app。
3. 只允许 `published` 状态下载。
4. 从 metadata 读取 `object_name`。
5. 生成 5 分钟有效 MinIO 预签名 URL。
6. 如果有完整性数据，响应头写 `X-App-SHA256`、`X-App-ETag`。
7. 记录下载热榜和持久化指标。
8. 返回 `307 Temporary Redirect` 到 MinIO 预签名 URL。

热榜链路：

- 下载成功后调用 `recordDownloadRank`。
- DB 中累加 `app_download_metrics`。
- Redis Pipeline 同时更新 daily、weekly、total 三个 ZSET。
- daily key 设置 48 小时 TTL，weekly key 设置 15 天 TTL，total 不设置 TTL。
- 查询热榜使用 `ZRevRangeWithScores`，再批量查 app 信息并过滤非 `published` 应用。

删除链路：

1. JWT 获取用户。
2. 查询 app 并校验开发者所有权。
3. 先删除 DB 记录。
4. 再删除 MinIO 对象。
5. 如果 MinIO 删除失败，写入 `minio_delete_retries` 补偿队列表。
6. 后台 worker 定时重试，最多 5 次，退避间隔为 `attempts^2` 分钟。

可讲亮点：

- 上传幂等键 + Redis 并发锁 + DB 唯一记录查询，解决重复发布和并发提交。
- 文件名、扩展名、文件头三重校验，不信任前端 `Content-Type`。
- MinIO 对象名服务端生成，避免路径穿越和对象覆盖。
- DB 失败时做对象存储补偿删除。
- 删除对象失败时有补偿队列和后台 worker。
- 下载使用预签名 URL，业务服务不直接代理大文件流量。
- Redis ZSET 做实时榜单，DB 指标做持久化。

雷点：

- 发布流程先上传 MinIO 再写 DB，不是强事务，所以必须讲清楚补偿机制。
- Redis 上传锁 TTL 是 2 分钟，如果上传或 DB 写入超过 TTL，极端情况下可能有并发窗口。
- `http.DetectContentType` 对部分压缩包可能返回 `application/octet-stream`，当前代码对 zip 做了兼容。
- metadata 是 map，读取时有类型断言，脏数据可能引发运行时问题。

## 10. Wallet 钱包调用链

实现位置：`services/wallet/router.go`、`services/wallet/app/http_service.go`、`services/wallet/repository/wallet_repository.go`。

接口：

- `GET /api/v1/wallets/me`：查询或创建当前用户钱包。
- `GET /api/v1/wallets/:wallet_id/transactions`：查询钱包流水。
- `POST /api/v1/wallets/:wallet_id/credit`：充值。
- `POST /api/v1/wallets/:wallet_id/debit`：扣款。
- `POST /api/v1/wallets/transfer`：转账。

钱包创建：

- `GetOrCreateByOwner` 先构造钱包，再用 GORM `OnConflict{DoNothing: true}` 插入。
- 插入后按 `owner_id + owner_type + currency_code` 查询返回。
- 适用于并发首次访问钱包的场景。

充值/扣款链路：

1. JWT 获取当前用户。
2. 校验 wallet_id 格式。
3. 查询钱包并校验 owner 是当前用户。
4. 金额用 `decimal.NewFromString` 解析，必须大于 0。
5. 进入 DB transaction。
6. 如果传了幂等键，先查是否已有流水。
7. 对钱包行加 `SELECT FOR UPDATE` 锁。
8. 再次检查幂等键，避免锁前后并发插入。
9. 领域对象执行 `Deposit` 或 `Deduct`。
10. 更新钱包余额。
11. 插入 `wallet_transactions` 流水。

转账链路：

1. 校验 from/to wallet 不相同，金额大于 0。
2. 进入 DB transaction。
3. 如果有幂等键，查 out/in 两条流水是否已存在。
4. 对两个钱包按 UUID 字符串排序后加锁，避免死锁。
5. 扣出账钱包，入账目标钱包。
6. 更新两个钱包余额。
7. 写两条流水，幂等键派生为 `{key}:out` 和 `{key}:in`。

可讲亮点：

- 金额使用 `shopspring/decimal`，避免 float 精度问题。
- 写操作都在 DB transaction 内完成。
- 钱包扣款使用行级锁 `FOR UPDATE`，避免并发扣成负数。
- 转账固定锁顺序，避免 AB/BA 死锁。
- 幂等键防重复充值、扣款、转账。

雷点：

- 幂等键必须有数据库唯一约束配合，否则极端并发下仅靠查询不能完全防重复；需要结合初始化 SQL 确认约束。
- 转账只有 from wallet 校验当前用户，to wallet 不要求当前用户所有，这符合转账语义但要明确。
- 金额响应统一 `StringFixed(4)`，数据库字段也是 `decimal(18,4)`，调用方不要传超过精度预期的小数。

## 11. Order 订单与购买调用链

实现位置：`services/order/router.go`、`services/order/app/http_service.go`、`services/order/repository/order_repository.go`。

接口：

- `POST /api/v1/orders/purchase`：一步购买。
- `POST /api/v1/orders`：创建订单。
- `GET /api/v1/orders`：订单列表。
- `GET /api/v1/orders/:order_id`：订单详情。
- `POST /api/v1/orders/:order_id/pay`：支付订单。
- `POST /api/v1/orders/:order_id/cancel`：取消订单。

创建订单链路：

1. JWT 获取当前用户。
2. 校验 app_id 和 amount。
3. 标准化币种，默认 `COIN`。
4. 如果有订单幂等键，先按幂等键查已存在订单。
5. 校验 app 存在且状态是 `published`。
6. 获取或创建用户钱包。
7. 创建 `pending` 订单。

支付链路：

1. 进入 DB transaction。
2. 对订单行加 `FOR UPDATE` 锁。
3. 校验订单归属当前用户。
4. 如果已支付，直接返回；若缺订阅则补发订阅。
5. 只允许 `pending` 订单支付。
6. 生成钱包扣款幂等键：`order:{order_id}:pay` 或 `order:{order_id}:pay:{idempotency_key}`。
7. 调用 WalletRepository 的 `DebitInTx`，在同一个事务内扣款并写流水。
8. 创建订阅 `subscriptions`，默认 1 年，`plan_type=purchase`。
9. 标记订单 paid，回填 `tx_id` 和 `subscription_id`。

一步购买链路：

- `Purchase` = `Create` + `Pay`。
- 适合前端直接购买应用。
- 拆分式创建/支付适合需要确认订单的流程。

可讲亮点：

- 订单、钱包扣款、流水、订阅发放放在同一个 DB transaction 内，保证一致性。
- 支付时锁订单行，防止重复支付并发竞争。
- 钱包扣款也锁钱包行，避免余额并发问题。
- 已支付订单再次支付返回已支付结果，具备一定幂等性。
- 支付后补发订阅逻辑能修复历史“已支付但订阅缺失”的不一致。

雷点：

- 当前订单金额由前端传入，文档也说明后续应迁移为后端从 app metadata 或 SKU/Price 表派生。
- `Purchase` 的创建和支付不是同一个外层事务，创建成功但支付失败时会留下 pending 订单，这是业务上可接受但要说明。
- 订单幂等键和支付幂等键语义不同，不能混用。

## 12. Admin 与 gRPC

实现位置：`common/grpcserver/server.go`、`services/admin/app/*`、`services/wallet/app/grpc_service.go`、`proto/*/v1`。

gRPC 启动：

- `grpcserver.New(database.DB)` 创建 gRPC server。
- 注册 `wallet.v1.WalletService`。
- 注册 `admin.v1.AdminQueryService`。
- 端口由 `grpc.port` 控制，默认 `12661`。

当前 gRPC 服务：

- Wallet：按 owner 查询钱包、查询或创建钱包。
- AdminQuery：查询虚拟账户、钱包流水、全局订单、变更日志。

可讲亮点：

- HTTP 面向前端和外部调用，gRPC 可面向内部服务间调用。
- proto 定义和生成代码分离，接口契约更稳定。
- Admin 查询通过 HTTP 和 gRPC 对齐字段，便于后续拆服务。

雷点：

- 当前 gRPC server 未看到认证拦截器，默认更适合内网调用；生产环境需要网关、mTLS 或 interceptor 鉴权。
- HTTP 与 gRPC 同进程，资源隔离有限，某一侧高负载可能影响另一侧。

## 13. 异步审计与补偿

实现位置：`common/audit/audit.go`。

审计 writer：

- `audit.Start(db)` 创建全局 writer。
- 内部 channel 容量 `1024`。
- `Emit` 非阻塞写入 channel，满了直接丢弃并打日志。
- 后台 goroutine 每 2 秒 flush 一次，或累计 100 条 flush。
- 批量写入 `audit_events`。
- 退出时关闭 channel，并在 shutdown timeout 内等待 flush。

MinIO 删除补偿 worker：

- 删除对象失败时写 `minio_delete_retries`。
- worker 每分钟扫描 `pending` 且到期的任务。
- 每次最多处理 50 条。
- 删除成功标记 `done`。
- 失败则 attempts + 1，最多 5 次。
- 下次执行时间按 `attempts * attempts` 分钟退避。

用到的 Go 并发知识点：

- `chan Event` 解耦业务线程和 DB 批量写。
- `select` 同时监听事件 channel 和 ticker。
- `context.WithTimeout` 控制 DB flush 和 MinIO 删除的最长耗时。
- `sync.Mutex` 保护 writer 的 closed 状态，避免关闭后继续写。
- `close(channel)` 通知 worker drain 剩余数据后退出。

雷点：

- 审计队列满了会丢事件，这是保护主链路的选择，不适合强审计合规场景。
- `Emit` 是进程内内存队列，进程崩溃会丢还没 flush 的事件。
- retry worker 是单进程内调度，多实例部署时要考虑重复扫描同一 retry 任务的问题，目前未看到任务锁或状态抢占。

## 14. 限流与请求池

限流实现位置：`common/middleware/rate_limit.go`、`common/middleware/redis_sliding_window.go`、`common/utils/token_bucket.go`。

请求池实现位置：`common/middleware/request_pool.go`、`common/utils/pool/gopool.go`。

请求池核心：

- 基于 `ants.NewPool(capacity, WithNonblocking(true))`。
- 默认容量来自配置 `features.request_pool.capacity`，当前是 `1000`。
- 每个 HTTP 请求提交到 request pool 执行。
- 池满时 `Submit` 返回 `ants.ErrPoolOverload`。
- 中间件返回 `503 SERVER_BUSY`，不会无限堆积请求。
- 请求完成后通过 `done := make(chan struct{})` 通知外层。

可讲亮点：

- 请求池相当于应用层背压，保护数据库、Redis、MinIO 等下游依赖。
- 非阻塞池实现快速失败，比无限 goroutine 或无限队列更可控。
- 退出时 `ReleaseTimeout` 等待池内任务完成，配合 HTTP server graceful shutdown。

雷点：

- 请求池适合控制并发，但会改变 Gin 正常执行模型。
- 池容量太小会导致误伤，太大又保护不了下游。
- 限流和请求池是两层机制：限流挡单用户或单 key 频率，请求池挡整体并发过载。

## 15. 数据一致性策略汇总

- DB transaction：钱包充值/扣款/转账、订单支付、订单取消。
- 行级锁：钱包行 `FOR UPDATE`，订单行 `FOR UPDATE`。
- 锁顺序：转账按 UUID 排序锁两个钱包，避免死锁。
- 幂等键：Market 发布、Wallet 写操作、Order 创建/支付。
- Redis 锁：Market 同一开发者同一幂等键并发上传互斥。
- 补偿机制：Market 发布 DB 失败删除 MinIO；Market 删除 MinIO 失败入补偿表重试。
- 缓存失效：Casbin 变更通过 RocketMQ 广播驱逐缓存。

面试可总结为：强一致部分尽量放同一个数据库事务内；跨存储系统的一致性用幂等、补偿和审计兜底。

## 18. 可以重点包装的项目亮点

- 混合认证：JWT + Redis Session，兼顾短 token 性能和服务端会话可控性。
- 分布式权限：Casbin domain RBAC + LRU Enforcer 缓存 + singleflight 防击穿 + RocketMQ 广播失效。
- 应用市场上传安全：文件名校验、扩展名校验、文件头校验、SHA256、ETag、服务端生成对象路径。
- 跨存储一致性：MinIO + DB 的非事务场景用补偿删除和审计兜底。
- 实时热榜：Redis ZSET + Pipeline + DB 持久化指标。
- 钱包并发安全：decimal 精确金额、DB transaction、行级锁、转账固定锁顺序。
- 订单支付一致性：订单锁、钱包锁、流水、订阅发放同事务。
- 应用层过载保护：ants 请求池非阻塞快速失败，避免请求无限堆积。
- 异步审计：channel + batch flush，降低主链路延迟。
- 优雅退出：HTTP/gRPC、worker、MQ、Redis、DB、协程池统一关闭。
- 双协议服务：REST 面向前端，gRPC 面向内部服务扩展。

## 20. 当前顶层风险与后续可优化点

- gRPC 缺少统一鉴权/限流拦截器，生产部署需要补齐。
- 请求池中在 goroutine 内执行 Gin context 是敏感设计，需要压测和 race 检查确认安全性。
- Market、Wallet、Order 的幂等能力需要结合 DB 唯一索引确认，否则并发极端场景仍有风险。
- MinIO retry worker 多实例部署时可能重复处理同一任务，需要任务抢占或状态 CAS。
- 订单金额由前端传入，后续应由后端价格体系派生。
- 审计事件是内存队列，进程崩溃会丢未 flush 事件，强审计场景可改 MQ 或 outbox。
- 权限中间件默认只从路由参数 `id` 取 domain，非群组资源需要额外资源所有者校验或扩展 domain 解析策略。
