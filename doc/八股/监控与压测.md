# 监控与压测八股与项目实战

本文基于当前项目实际部署脚本、监控配置和 k6 脚本整理，覆盖 Prometheus、cAdvisor、Loki、Alloy、Grafana、k6、日志 trace、Docker Compose 资源限制和面试拷打点。

## 1. 项目监控压测总览

| 能力 | 实际文件 | 组件 |
| --- | --- | --- |
| 容器编排 | `script/docker-compose.yaml` | Docker Compose |
| 指标采集 | `script/prometheus.yml` | Prometheus |
| 容器指标 | compose 中 `cadvisor` | cAdvisor |
| 日志存储 | `script/loki-config.yaml` | Loki + MinIO S3 |
| 日志采集 | `script/alloy-config.alloy` | Grafana Alloy |
| 可视化 | `script/grafana/provisioning/*` | Grafana |
| 压测执行 | `script/run-load-tests.sh`、`script/k6-scripts/*` | k6 |
| 压测指标写入 | compose 中 k6 env | Prometheus remote write |
| 容器资源采样 | `run-load-tests.sh` | `docker stats` CSV |
| 应用请求日志 | `common/middleware/trace.go`、`common/utils/logs/loger.go` | zap JSON stdout -> Alloy -> Loki |

## 2. Docker Compose 里的监控栈

位置：`script/docker-compose.yaml`

服务列表：

- `prometheus`：指标数据库和查询。
- `cadvisor`：采集 Docker 容器 CPU、内存、网络、磁盘等指标。
- `loki`：日志存储。
- `grafana`：指标和日志可视化。
- `alloy`：发现 Docker 容器并采集 stdout 日志到 Loki。
- `k6`：压测容器，profile 为 manual，不会默认启动。

App 容器资源限制：

```yaml
app:
  deploy:
    resources:
      limits:
        cpus: '2.0'
        memory: 2G
      reservations:
        cpus: '2.0'
        memory: 2.0G
```

注意：

- Docker Compose 的 `deploy.resources` 在不同 compose/docker 运行模式下支持程度要确认。
- 注释里写“最高允许 1 个 CPU/512MB”等和实际值 `2.0`、`2G` 不一致，面试或文档中按实际配置说。

## 3. Prometheus 配置

位置：`script/prometheus.yml`

```yaml
global:
  scrape_interval: 15s
  evaluation_interval: 15s

scrape_configs:
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]

  - job_name: "cadvisor"
    static_configs:
      - targets: ["cadvisor:8080"]
```

当前实际采集：

- Prometheus 自身指标。
- cAdvisor 暴露的容器指标。
- Go 应用 `/metrics` 端点。
- k6 通过 Prometheus remote write 推送的压测指标。
- 应用 HTTP 请求总数、HTTP 响应耗时 Histogram、GORM 连接池状态和请求池拒绝总数。
- Go runtime 默认 collector 暴露的 goroutine 等运行时指标。

当前仍未覆盖：

- Redis、MinIO、RocketMQ 内部指标 exporter。
- OpenTelemetry 标准 trace/span。
- 订单数、下载数等更细的业务计数指标。

Prometheus 启动参数：`docker-compose.yaml`

```yaml
--web.enable-remote-write-receiver
```

作用：

- 允许 k6 通过 Prometheus remote write 把压测指标写进 Prometheus。

面试拷打：

- Pull 模型：Prometheus 定时 scrape target。
- Remote write：其他系统把指标推到 Prometheus remote write receiver，当前 k6 使用这个能力。
- scrape interval 是 15 秒，短压测时要注意指标粒度。

雷点：

- 应用内部基础指标已经接入，但 DB/Redis/MinIO/RocketMQ 依赖侧仍缺少独立 exporter 视角。
- 如果压测时间很短，15 秒 scrape 可能错过峰值。

## 4. cAdvisor 容器指标

compose 配置：

```yaml
cadvisor:
  image: gcr.io/cadvisor/cadvisor:v0.55.1
  container_name: RPA-cadvisor
  ports:
    - "8082:8080"
  volumes:
    - /:/rootfs:ro
    - /var/run:/var/run:rw
    - /sys:/sys:ro
    - /var/lib/docker/:/var/lib/docker:ro
  privileged: true
  devices:
    - /dev/kmsg:/dev/kmsg
```

cAdvisor 能看：

- 容器 CPU 使用率。
- 容器内存使用量。
- 网络 IO。
- block IO。
- 容器生命周期和标签。

项目压测脚本还额外采集 `docker stats`：

```bash
docker stats --no-stream --format "timestamp,name,cpu_perc,mem_usage,mem_perc,net_io,block_io,pids"
```

为什么两套都要：

- Prometheus/cAdvisor 适合图表和长期趋势。
- `docker stats` CSV 适合每次压测结果归档，和 k6 summary 放同一目录。

雷点：

- cAdvisor 采的是容器层，不知道业务接口、SQL 慢查询、Redis 慢命令。
- 容器 CPU 百分比要结合 CPU limit 看，不同环境解释不同。

## 5. Loki 日志存储

位置：`script/loki-config.yaml`

核心配置：

```yaml
auth_enabled: false
server:
  http_listen_port: 3100
common:
  ring:
    kvstore:
      store: memberlist
  storage:
    s3:
      endpoint: minio:9000
      insecure: true
      bucketnames: loki-bucket
      access_key_id: ${MINIO_ROOT_USER}
      secret_access_key: ${MINIO_ROOT_PASSWORD}
      s3forcepathstyle: true
  replication_factor: 1
schema_config:
  configs:
    - from: 2024-01-01
      store: tsdb
      object_store: s3
      schema: v13
```

项目实际设计：

- Loki 单机模式。
- 对象存储用 MinIO 的 `loki-bucket`。
- `config.expand-env=true` 让 Loki 使用 compose 注入的 MinIO 用户密码。
- `auth_enabled: false`，本地环境无鉴权。

MinIO bucket 创建：`docker-compose.yaml`

```sh
mc mb myminio/loki-bucket --ignore-existing
```

雷点：

- Loki `auth_enabled: false` 适合本地，不适合公网生产。
- `replication_factor: 1` 是单副本，节点或 MinIO 数据丢失会影响日志。
- Loki 适合日志检索，不适合高基数字段索引。trace_id 可以作为日志字段查询，但 label 设计要控制基数。

## 6. Alloy 日志采集

位置：`script/alloy-config.alloy`

链路：

1. `discovery.docker "containers"` 发现 Docker 容器。
2. `discovery.relabel "docker_logs"` 清洗标签。
3. `loki.source.docker "log_collect"` 采集容器日志。
4. `loki.write "local_loki"` 推送到 `http://loki:3100/loki/api/v1/push`。

关键标签：

- `container`：优先使用 compose service 名。
- `image`：容器镜像。
- `job=local_docker`。
- `namespace=docker`。
- `app`：复制 container。
- `instance`：复制 container。

compose 挂载：

```yaml
alloy:
  volumes:
    - ./alloy-config.alloy:/etc/alloy/config.alloy:ro
    - /var/run/docker.sock:/var/run/docker.sock:ro
    - /var/log:/var/log:ro
    - /var/lib/docker/containers:/var/lib/docker/containers:ro
```

为什么需要 Docker socket：

- Alloy 通过 Docker API 发现容器和元数据。

雷点：

- 挂 Docker socket 有安全风险，本地可接受，生产要谨慎。
- Alloy 只采 stdout/stderr，所以应用日志必须写标准输出。
- 当前 zap logger 正是写 stdout。

## 7. 应用日志和 Trace ID

日志初始化：`common/utils/logs/loger.go`

```go
encoderConfig := zap.NewProductionEncoderConfig()
encoderConfig.EncodeTime = zapcore.ISO8601TimeEncoder
core := zapcore.NewCore(
    zapcore.NewJSONEncoder(encoderConfig),
    zapcore.AddSync(os.Stdout),
    zap.NewAtomicLevelAt(zap.InfoLevel),
)
Log = zap.New(core, zap.AddCaller(), zap.AddStacktrace(zap.ErrorLevel))
```

特点：

- JSON 日志。
- ISO8601 时间。
- 输出 stdout，供 Alloy 采集。
- error 级别带 stacktrace。

请求日志：`common/middleware/trace.go`

```go
logs.Log.Info("API_REQUEST",
    zap.Int("status", c.Writer.Status()),
    zap.String("method", c.Request.Method),
    zap.String("path", path),
    zap.String("query", query),
    zap.String("ip", c.ClientIP()),
    zap.String("trace_id", traceID),
    zap.Duration("latency", latency),
)
```

Trace ID：`common/response/response.go`

```go
traceID := strings.TrimSpace(c.GetHeader("X-Trace-ID"))
if traceID == "" {
    traceID = uuid.NewString()
}
c.Set(TraceIDKey, traceID)
c.Header("X-Trace-ID", traceID)
```

错误响应：

```json
{"code":"...","message":"...","request_id":"trace-id"}
```

压测脚本配合：`script/k6-scripts/lib/common.js`

```js
headers: Object.assign({ 'X-Trace-ID': `k6-${name}-${uniqueID('trace')}` }, headers)
```

亮点：

- 每个请求都能通过 `X-Trace-ID` 串联客户端、API 日志和错误响应。
- k6 压测显式带 trace id，便于在 Loki 中定位单个压测请求。

雷点：

- 当前 trace id 只是请求级 ID，不是 OpenTelemetry trace/span。
- `logs.WithContext` 从 ctx 中取 `trace_id`，但 `response.TraceID` 存在 Gin context，不一定写入 `request.Context()`，所以 `WithContext` 目前未必能拿到 Gin trace id。

## 9. k6 压测容器

compose 配置：

```yaml
k6:
  image: grafana/k6:latest
  profiles:
    - manual
  volumes:
    - ./k6-scripts:/scripts
  environment:
    - K6_PROMETHEUS_RW_SERVER_URL=http://prometheus:9090/api/v1/write
    - K6_PROMETHEUS_RW_TREND_AS_NATIVE_HISTOGRAM=false
    - K6_PROMETHEUS_RW_TREND_STATS=p(90),p(95),p(99),min,max,avg,med
```

特点：

- profile 是 `manual`，平常 `docker compose up -d` 不会自动跑压测。
- 压测脚本挂载在 `/scripts`。
- k6 指标 remote write 到 Prometheus。
- trend 指标包含 p90、p95、p99、min、max、avg、med。

运行封装：`script/run-load-tests.sh`

```bash
docker compose run --rm k6 run \
  -o experimental-prometheus-rw \
  --tag "testid=${name}" \
  --summary-export "/scripts/results/.../${name}.summary.json" \
  "${script}"
```

输出：

- `${name}.summary.json`
- `${name}.log`
- `${name}.exitcode`
- `${name}.docker-stats.csv`
- `manifest.txt`

## 10. k6 公共库设计

位置：`script/k6-scripts/lib/common.js`

默认环境变量：

- `BASE_URL`
- `EMAIL`
- `PASSWORD`
- `AUTH_MODE`
- `APP_ID`
- `APP_FILE`
- `AVATAR_FILE`

单接口压测 options：

```js
executor: 'constant-arrival-rate',
rate: Number(__ENV.RATE || 50),
timeUnit: __ENV.TIME_UNIT || '1s',
duration: __ENV.DURATION || '1m',
preAllocatedVUs: Number(__ENV.VUS || 100),
maxVUs: Number(__ENV.MAX_VUS || 200)
```

阈值：

```js
http_req_failed: ['rate<0.05']
http_req_duration: ['p(95)<1500']
checks: ['rate>0.95']
```

鉴权模式：

- `AUTH_MODE=bypass`：不登录，适合配合后端 `FEATURES_JWT_AUTH=false`。
- `AUTH_MODE=login`：setup 阶段登录，带 cookies。

请求标签：

```js
tags: { name }
```

作用：

- k6 指标里可以按接口名聚合。
- `--tag testid=${name}` 可以按测试批次过滤。

## 11. run-load-tests.sh 调用链

位置：`script/run-load-tests.sh`

主要步骤：

1. 创建结果目录。
2. 创建 k6 fixture：`app.gz`、`avatar.png`。
3. 写 `manifest.txt`，记录功能开关、base URL、compose 版本等。
4. `restart_app` 按功能开关重建 app 容器。
5. `ensure_load_user` 注册压测用户，并用 SQL 授予 superadmin。
6. 先跑 smoke。
7. 跑 HTTP API suite。
8. 可选跑 MQ suite。
9. 每个 k6 case 同时启动 `docker stats` 采样。

功能开关：

```bash
FEATURES_JWT_AUTH
FEATURES_CASBIN_AUTHZ
FEATURES_RATE_LIMIT_ENABLED
FEATURES_REQUEST_POOL_ENABLED
```

压测时默认把 rate limit/request pool 容量调大：

```bash
FEATURES_RATE_LIMIT_RATE=100000
FEATURES_RATE_LIMIT_CAPACITY=100000
FEATURES_RATE_LIMIT_LIMIT=100000
FEATURES_REQUEST_POOL_CAPACITY=10000
```

目的：

- 隔离业务接口性能，不让限流和请求池成为瓶颈。
- 如果要专门测限流/请求池，需要反过来调小阈值。

雷点：

- 脚本直接用 `docker exec RPA-pgdb psql ...` 给压测用户授权，适合本地，不是生产流程。
- `restart_app` 会 `--force-recreate app`，压测前要确认不会影响其他人。
- 压测结果目录默认带时间戳，适合归档，但要注意不要无限提交历史结果。

## 15. 如何看一次压测结果

结果目录示例：`script/k6-scripts/results/20260619-225054/`

关键文件：

- `manifest.txt`：本次测试环境和开关。
- `*.summary.json`：k6 汇总指标。
- `*.log`：k6 输出日志。
- `*.exitcode`：k6 退出码。
- `*.docker-stats.csv`：压测期间容器资源采样。

分析顺序：

1. 看 `exitcode` 是否非 0。
2. 看 `summary.json` 的 `http_req_failed`、`http_req_duration`、`checks`。
3. 看接口 p95/p99 是否超过阈值。
4. 看 `docker-stats.csv` 中 app、db、redis、minio CPU/内存是否打满。
5. 去 Grafana/Prometheus 看同一 `testid` 的 k6 指标趋势。
6. 去 Loki 用 `trace_id` 或容器名查错误日志。

## 16. 常见定位场景

### 16.1 p95 高但 CPU 不高

可能原因：

- DB 锁等待。
- Redis/MinIO 网络或连接池等待。
- 请求池容量限制。
- 外部依赖慢。

项目对应排查：

- Wallet/Order 看 DB 行锁和事务。
- Market 发布看 MinIO 上传和 DB 写入。
- JWT 接口看 Redis Session。

### 16.2 429 多

可能原因：

- `FEATURES_RATE_LIMIT_ENABLED=true`。
- memory rate/capacity 太小。
- Redis sliding window limit 太小。

项目对应：公开接口默认挂 `ConfiguredRateLimit()`。

### 16.3 503 多

可能原因：

- 请求池满，返回 `SERVER_BUSY`。
- 依赖不可用。

项目对应：`ConfiguredRequestPoolFastFail()` 全局中间件，池满返回 `503`。

### 16.4 登录接口慢

可能原因：

- bcrypt 密码校验 CPU 成本。
- Redis Session 写入慢。
- DB 按 email 查用户慢。

### 16.5 Market 发布慢

可能原因：

- multipart 上传和临时文件写入。
- 文件头检测和 SHA256。
- MinIO 上传。
- DB insert 和唯一索引检查。
- 审计 channel 一般不阻塞，除非日志/DB异常只影响异步审计。

## 17. 面试拷打点

### 17.1 Prometheus Pull vs Push

项目里 Prometheus 对 cAdvisor 是 pull scrape；k6 指标通过 remote write 推给 Prometheus。要明确这两条链路不同。

### 17.2 RED / USE 指标

服务接口看 RED：

- Rate：请求速率。
- Errors：错误率。
- Duration：延迟。

资源看 USE：

- Utilization：CPU/内存/磁盘利用率。
- Saturation：队列、连接池等待、CPU throttling。
- Errors：容器重启、IO 错误等。

项目当前：

- k6 提供接口 RED。
- cAdvisor 提供容器 USE 的一部分。
- 应用内部连接池/DB 慢查询指标还不足。

### 17.3 p95 / p99 为什么重要

平均值会掩盖尾延迟。压测要看 p95/p99，项目 k6 已配置 p90/p95/p99 remote write。

### 17.4 constant-vus vs constant-arrival-rate

- constant-vus：固定并发用户数，吞吐由响应速度决定。
- constant-arrival-rate：固定到达率，更适合测服务在指定 RPS 下的表现。

项目单接口压测默认使用 constant-arrival-rate。

### 17.5 日志和指标的区别

- 指标适合聚合趋势和告警。
- 日志适合排查具体错误上下文。
- trace id 用来把一次请求串起来。

项目实际：API_REQUEST JSON 日志进入 Loki，k6/容器指标进入 Prometheus。

## 18. 当前监控压测风险和优化点

- 应用 `/metrics` 已接入 Prometheus client，后续可继续补业务计数、Redis pool stats、依赖侧 exporter 和告警规则。
- Prometheus 当前只 scrape cAdvisor 和自己，无法直接看 DB/Redis/MinIO/RocketMQ 内部指标。
- Loki 单副本 + auth disabled 只适合本地。
- Alloy 挂 Docker socket 有安全风险。
- `logs.WithContext` 与 Gin trace id 尚未完全贯通到 request context。
- k6 结果目录已有大量历史结果，提交仓库时要控制体积。
- 压测脚本默认重建 app 容器，不适合多人共享环境随意运行。
- `docker stats` 是采样值，不能替代完整 profiling。
- 如果要定位 Go CPU/内存热点，还需要 pprof。

## 19. 可以继续补的高级能力

- 业务指标：订单创建/支付数、下载成功数、权益发放成功/失败数。
- Redis pool stats：go-redis `PoolStats()`。
- pprof：CPU、heap、goroutine、mutex、block profile。
- OpenTelemetry：真正分布式 trace，替代单一 `X-Trace-ID`。
- Alertmanager：错误率、p95、容器重启、CPU/内存、Redis/DB 不可用告警。
- k6 thresholds 按接口标签细分，而不是只看全局。
