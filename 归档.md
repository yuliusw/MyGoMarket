# RPA Market 已完成修复归档

来源：`ai-task.md`。本文只保留已修复、已完成、可用于面试复习的重点。

## 1. 项目总览

`RPA-market` 是 Go + Vue 的 RPA 应用市场单体仓库。

技术栈：Gin、GORM、PostgreSQL、Redis、MinIO、RocketMQ、Casbin、Vue 3、Vite、TypeScript。

核心模块：

- IAM：注册、登录、用户资料、群组、RBAC。
- Market：应用发布、更新、下架、删除、下载、热榜。
- Common：配置、数据库、Redis、MinIO、JWT、Casbin、限流、审计、响应封装。
- Wallet：钱包领域雏形和 gRPC 基础接口。

启动链路：

```text
config -> GORM -> Redis -> MinIO -> RocketMQ -> Casbin -> IAM 路由 -> Market 路由
```

## 2. 认证与 Session

已完成：

- JWT 只允许 `HS256`，避免签名算法混淆风险。
- 登录成功前必须写入 Redis Session，失败返回 `503`。
- 登出、改密会清理 `auth_token` 和 `session_id` Cookie。
- JWTAuth 同时校验 JWT 与 Redis Session。
- 旧 `GetUserCookie/DeleteUserCookie` 已统一为 `GetSession/DeleteSession`。

当前模型：

- `auth_token`：短 JWT，约 30 分钟。
- `session_id`：长 Session，Redis 保存约 7 天。
- Redis key：`user:session:<userID>`。

解决的问题：

- JWT 无法主动失效。
- 登出、改密后旧 Token 仍可用。
- Redis 写 Session 失败却返回登录成功，导致伪登录。

八股考点：

- JWT 无状态和 Session 有状态的取舍。
- 单点登录、顶号、登出失效怎么做。
- Cookie、HttpOnly、Bearer Token 区别。
- JWT 为什么要限制签名算法。

面试说法：

```text
我没有只用 JWT，而是做了 JWT + Redis Session。JWT 负责短期身份声明，Redis Session 负责服务端主动失效。这样登出、改密、顶号都能立即生效。
```

## 4. 限流与过载保护

已完成：

- 内存令牌桶限流：适合单实例本地压测。
- Redis ZSET + Lua 滑动窗口限流：适合多实例。
- 登录、注册、Market 列表、详情、下载、热榜已挂限流。
- 全局 Gin 链路接入请求池，池满返回 `503` 快速失败。
- DB/Redis 连接池和 GORM 日志级别配置化。

解决的问题：

- 防止热点接口被刷爆。
- 防止请求无限排队拖垮服务。
- 多实例下共享限流状态。

八股考点：

- 令牌桶、漏桶、固定窗口、滑动窗口。
- Redis ZSET 排序集合。
- Lua 原子性。
- 快速失败 vs 排队等待。

面试说法：

```text
限流做了 memory 和 redis 两套。单机用令牌桶，多实例用 Redis ZSET + Lua 滑动窗口。服务过载时请求池直接 503，避免请求堆积导致雪崩。
```

## 5. Casbin 权限与缓存热更新

已完成：

- Casbin model 路径配置化：`config/casbin/RBAC.conf`。
- 使用 domain RBAC，支持团队维度权限隔离。
- Casbin Manager 使用 LRU + singleflight 懒加载 domain enforcer。
- 增加 TTL 兜底，默认 600 秒。
- 权限变更后先本地失效，再通过 RocketMQ 广播。
- 成员邀请、踢出、退出、解散、角色修改会清理 domain 缓存。
- 角色权限全量替换会 purge 全部缓存。
- 初始化权限已补齐：应用创建、编辑、下架、删除，群组查看、编辑、删除、邀请、踢人，角色管理。

解决的问题：

- 每次鉴权查 DB 性能差。
- 多实例本地缓存不一致。
- 权限变更后旧缓存导致越权或权限不生效。

八股考点：

- RBAC、ABAC、ACL。
- Casbin `sub/dom/obj` 模型。
- singleflight 防缓存击穿。
- 本地缓存 + MQ 失效广播。
- TTL 兜底。

面试说法：

```text
权限使用 Casbin domain RBAC。为了避免每次鉴权查 DB，我做了 LRU 缓存和 singleflight 懒加载。权限变化时先清本地缓存，再用 RocketMQ 通知其他实例清缓存，TTL 做兜底。
```

注意：Market 当前 app 权限有全局域 fallback；资源级控制已通过 `developer_id` 所有者校验补充在更新、下架、删除接口上。

## 6. Market 上传链路

已完成：

- 发布接口支持 multipart 上传 `app_file`。
- 文件大小限制 100MB。
- 文件先落 `/tmp`，再上传 MinIO。
- MinIO object name 改成服务端 UUID 路径：`apps/{developer_id}/{app_id}/app{ext}`。
- 原始文件名只进 metadata，允许中文，但拒绝路径分隔符、NUL、非法 UTF-8。
- 文件类型白名单：`.zip`、`.gz`、`.tgz`，并校验文件头。
- metadata 保存 `object_name`、`sha256`、`etag`、`file_name`。
- 支持 `Idempotency-Key` / `idempotency_key`。
- Redis 短锁 + DB 唯一索引兜底，防止重复发布。
- 并发冲突返回 `409`，重复幂等返回已有 `app_id`。
- DB 失败或超时会补偿删除 MinIO 对象，并写审计。
- 头像上传也已改为 UUID 文件名，并增加扩展名和内容 sniff。

解决的问题：

- 重复点击、网络重试导致重复发布。
- 并发上传产生脏数据。
- DB 失败后 MinIO 残留孤儿文件。
- 文件名路径注入和伪造扩展名风险。

八股考点：

- 幂等键设计。
- Redis 锁和 DB 唯一索引双保险。
- 文件上传安全。
- 对象存储和数据库最终一致性。
- 补偿事务。

面试说法：

```text
上传链路用 Idempotency-Key 防重复提交。Redis 短锁挡并发，DB 唯一索引做最终兜底。文件先上传 MinIO，再写 DB；如果 DB 失败，就异步补偿删除对象，避免孤儿文件。
```

## 7. Market 下载与热榜

已完成：

- 只允许 `published` 应用下载。
- 从 metadata 读取 `object_name`。
- 生成 5 分钟 MinIO 预签名 URL。
- 返回 `307 redirect`，业务服务不转发大文件。
- 响应带 `X-App-SHA256`、`X-App-ETag`。
- 下载成功后写 Redis ZSET 热榜。
- 下载指标同步落库到 `app_download_metrics`。

Redis 热榜：

- 日榜：`market:rank:downloads:daily:<yyyymmdd>`，TTL 48h。
- 周榜：`market:rank:downloads:weekly:<yyyyww>`，TTL 15d。
- 总榜：`market:rank:downloads:total`，不过期。

解决的问题：

- 大文件下载不占用业务服务带宽。
- 实时排行用 Redis，长期统计用 DB。
- 客户端可校验文件完整性。

八股考点：

- S3/MinIO 预签名 URL。
- 307 重定向。
- Redis ZSET 排行榜。
- 实时数据与持久化数据分层。

面试说法：

```text
下载时服务端只做状态校验、生成短期预签名 URL 和记录统计，然后 307 跳转到 MinIO。热榜用 Redis ZSET 实时维护，同时写 DB 指标表做长期统计。
```

已完成：

- `UpdateApp` 改成字段级 `Updates`，避免 `Save` 全字段覆盖。
- `OffShelfApp` 改成字段级状态更新。

- GORM `Save` 可能造成 lost update。
- 对象存储删除失败需要最终一致性补偿。
- 审计日志不能阻塞主链路，也不能退出时丢失。

八股考点：

- Lost Update。
- 字段级更新、乐观锁、悲观锁。
- 最终一致性和补偿队列。
- Channel + Ticker 批量消费。
- 优雅退出时 flush。

面试说法：

```text
我把 Save 改成字段级 Updates，避免旧结构体覆盖并发更新。删除应用时先删 DB，再删 MinIO；如果 MinIO 删除失败，就写补偿表，由 worker 按 next_run_at 重试，最多 5 次后标记失败。
```

## 14. 高频追问速记

### JWT 为什么还要 Redis Session？

JWT 无状态，不能主动失效；Redis Session 可以支持登出、改密、顶号。代价是每次请求需要查 Redis。

### 幂等为什么同时用 Redis 锁和 DB 唯一索引？

Redis 锁减少并发冲突，DB 唯一索引做最终兜底。Redis 可能过期或异常，但数据库约束不会绕过。

### 文件上传 DB 失败怎么办？

MinIO 上传成功但 DB 失败会产生孤儿对象，所以失败时用 cleanup context 删除对象；删除失败则进入补偿队列重试。

### 为什么下载用预签名 URL？

业务服务不承载大文件流量，只负责鉴权、校验、统计和生成短期 URL，文件传输交给 MinIO。

### Redis ZSET 怎么做热榜？

app_id 作为 member，下载次数作为 score。下载时 `ZINCRBY`，查询时 `ZREVRANGE WITHSCORES`。日榜、周榜设置 TTL，总榜不过期。

### GORM Save 有什么问题？

`Save` 可能把旧结构体的所有字段写回，覆盖其他请求刚更新的字段。改用字段级 `Updates` 只更新本次提交字段。

### 权限缓存怎么保持多实例一致？

本地 LRU 缓存提升性能；权限变更时清本地缓存，并通过 RocketMQ 广播其他实例清缓存；TTL 处理消息丢失兜底。

### 钱包扣款怎么防止负余额？

不能读余额后应用层判断再保存。应使用事务 + `SELECT FOR UPDATE` 锁行，或 `UPDATE ... WHERE balance >= amount` 原子扣减。

### 为什么现在不拆微服务？

当前单体还可控，先用模块分层和 gRPC 契约稳定边界。过早拆分会引入服务发现、网络重试、分布式事务等复杂度。
